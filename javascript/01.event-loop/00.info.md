# The Event Loop & Macrotasks/Microtasks

## The Event Loop

To summarize as an expert developer, the **Event Loop** is the coordinator that manages how your code executes, ensuring that synchronous tasks run immediately and asynchronous tasks wait their turn without freezing the main thread.

### 🟢 The 4 Key Pillars

1. **Single-Threaded Execution:** JavaScript has one **Call Stack**. It can only do one thing at a time. If you run a massive loop, you "block the loop," and the UI freezes.

2. **Offloading to the Environment:** When you call an async function (like `fetch` or `setTimeout`), JS doesn't wait. It hands the task to the browser (Web APIs) or Node.js runtime and moves to the next line of code.

3. **The Priority Hierarchy:** Not all "waiting" tasks are equal.

   - **Microtasks (Promises, `queueMicrotask`):** High priority. The loop will not move on until the Microtask queue is completely empty.
   - **Macrotasks (`setTimeout`, `setInterval`, I/O):** Lower priority. Only one macrotask is processed per loop cycle.

4. **The "Check" Mechanism:** The Event Loop’s only job is to look at the **Call Stack**. If the stack is **empty**, it pushes the first available task from the queues (starting with Microtasks) onto the stack to be executed.

### 💡 The "Expert" Mental Model

Think of it like a **Restaurant**:

- **The Call Stack** is the Chef. He can only cook one dish at a time.
- **The Web APIs** are the Ovens/Waiters. They handle the long-running tasks (baking, delivering) in the background.
- **The Microtask Queue** is an "Emergency Order." The Chef must finish all emergency orders before looking at the regular line.
- **The Macrotask Queue** is the "Regular Line" of customers waiting for their food.

### The Execution Order Rule

> **Sync Code** → **All Microtasks** → **One Macrotask** → **Repeat.**

For an expert developer, understanding the distinction between **Macrotasks** and **Microtasks** is essentially understanding the "heartbeat" of JavaScript performance and bug prevention.

## Macrotasks/Microtasks

While both handle asynchronous callbacks, the engine treats them with different levels of urgency.

#### **1. Microtasks (High Priority)**

- **Sources:** `Promise.then/catch/finally`, `await`, `queueMicrotask()`, and `MutationObserver`.
- **Execution Timing:** They run **immediately after** the current synchronous code finishes and **before** the browser renders or moves to the next macrotask.
- **The "Emptying" Rule:** The Event Loop will not move to the next macrotask until the Microtask Queue is **completely empty**. If a microtask schedules another microtask, that one runs too (this can lead to "starving" the event loop if not careful).

#### **2. Macrotasks (Low Priority)**

- **Sources:** `setTimeout`, `setInterval`, `setImmediate` (Node), I/O, and UI rendering.
- **Execution Timing:** They are picked from the queue one at a time. Between each macrotask, the browser may re-render the page.
- **The "One-by-One" Rule:** Only **one** macrotask is processed per loop iteration. After it finishes, the engine immediately checks for new microtasks before looking at the next macrotask.

### 📋 Comparison Table

| Feature       | Microtasks                        | Macrotasks (Tasks)              |
| ------------- | --------------------------------- | ------------------------------- |
| **Examples**  | Promises, `await`                 | `setTimeout`, `fetch` callbacks |
| **Priority**  | Highest                           | Lowest                          |
| **Execution** | Entire queue is cleared per cycle | Only one per cycle              |
| **Render**    | Happens _after_ queue is cleared  | Happens _between_ tasks         |

### 🛠️ Practical "Expert" Insight

If you want to update the UI or ensure a heavy calculation doesn't block user input, use a **Macrotask** (`setTimeout`) to "break up" the work.

If you need to ensure a piece of logic runs immediately after a state change but before the user sees anything on screen, use a **Microtask** (`Promise.resolve()`).
